# Kaiper AI — 기능 명세 및 개발자 코멘트

---

## 1. 사내 문서 검색 (Knowledge Search)

**무엇을 하는가**
ChromaDB 벡터 DB에 인제스트된 사내 문서에서 사용자 질문에 맞는 내용을 검색하고 LLM이 요약·답변한다.

**흐름**
`search_node` → `quality_check_node` → (품질 OK면) `answer_node` / (문서 없으면) `rewrite_node` → `search_node` (최대 2회)

**개발자 코멘트**

> 단순 벡터 검색만으로는 정확한 키워드가 포함된 문서를 놓치는 경우가 있다. 그래서 벡터 유사도(Chroma/Gemini 임베딩) + BM25 키워드 검색을 병렬 수행한 뒤, **Reciprocal Rank Fusion(RRF)** 알고리즘으로 두 결과를 합산했다. 단순 점수 합산이 아닌 RRF를 쓴 이유는 각 검색 방식의 점수 스케일이 달라 직접 비교가 불가능하기 때문이다.
>
> 검색 결과가 없을 때 바로 "못 찾았다"고 답하면 UX가 나빠진다. **query rewrite loop**를 도입해 LLM이 쿼리를 더 범용적인 키워드로 재작성하고 재검색하게 했다. 무한루프 방지를 위해 최대 2회 제한 후 강제 통과 처리한다.

---

## 2. 심층 검색 (Detail Search)

**무엇을 하는가**
"더 자세히 알려줘", "그 부분 좀 더" 같은 후속 질문을 받으면, 직전 Q&A 쌍을 재구성해 보다 정밀한 검색을 수행한다.

**흐름**
`task_router` (detail 힌트 감지) → `detail_search_node` → `answer_node`

**개발자 코멘트**

> 사용자가 "더 자세히"라고만 하면 task_router 입장에서는 무엇에 대한 질문인지 알 수 없다. `_reconstruct_query()`는 직전 AI 답변에서 주제를 추출해 검색 쿼리를 재구성한다. 또한 직전 답변에서 언급된 문서명을 파싱해 **source 필터**를 걸어 동일 문서 내에서 k=15로 더 넓게 검색한다. source 필터 결과가 없으면 필터 없이 재시도한다.

## 3. 이메일 초안 작성 (Email Draft)

**무엇을 하는가**
수신자·제목·본문·참조(CC)를 포함한 업무 이메일 초안을 JSON 형태로 생성한다. 작성 후 사용자 승인 게이트를 거친다.

**흐름**
`task_router` → (수신자 미확인 시) `clarification` → `email_draft_node` → `human_review_node`

**개발자 코멘트**

> 이메일 작성에서 가장 자주 발생하는 작업은 "수신자만 바꿔줘", "제목 수정해줘" 같은 부분 수정이다. LLM을 다시 돌리는 것은 과잉 처리이므로, 수정 키워드 + 변경 필드가 명확하면 **결정론적 패치(deterministic patch)** 경로로 분기해 해당 필드만 교체한다. 이 경로를 먼저 시도하고, 해당하지 않을 때만 LLM을 호출한다.
>
> 수신자 추론은 세 단계로 한다: 입력 텍스트에서 이메일 주소 직접 추출 → 부서명 매핑 테이블(`DEPT_EMAIL_MAP` 환경변수로 외부 주입 가능) → 슬롯 미충족이면 clarification으로 물어본다. LLM이 이전 대화의 수신자를 그대로 가져오는 오염 문제를 막기 위해 "현재 요청에서만 수신자를 파악하라"는 지시를 시스템 프롬프트에 명시했다.

## 4. RFP 초안 작성 (RFP Draft)

**무엇을 하는가**
제안요청서(RFP) 전체 문서를 작성한다. 사내 문서 사전 조사 → 초안 작성 → AI 자체 검토 → 사용자 승인의 4단계 파이프라인으로 진행된다.

**흐름**
`task_router` → `rfp_research_node` → `rfp_draft_node` → `rfp_review_node` → (검토 통과 시) `human_review_node` / (미흡 시) `rfp_draft_node` (최대 1회 재작성)

**개발자 코멘트**

> 단일 LLM 호출로 RFP를 한 번에 쓰면 섹션 누락이 잦다. **research → draft → review** 3단계로 분리한 이유는 각 단계의 역할을 명확히 하고, 검토 단계에서 객관적인 시각으로 품질을 판단하기 위해서다. "같은 모델이 쓰고 검토해봤자 의미 없다"는 반론이 있을 수 있지만, 역할 분리 프롬프트 자체가 다른 평가 기준을 유도한다.
>
> research 단계에서 `_resolve_search_query()`를 쓰는 이유는 "이걸로 RFP 작성해줘"처럼 모호한 지시어가 왔을 때 직전 AI 답변에서 주제를 뽑아 검색 쿼리를 만들기 위해서다. 재검토 루프는 최대 1회로 제한했다. 실무 환경에서 LLM 품질 루프를 너무 깊게 돌리면 레이턴시 문제가 생기고 사용자 체감이 나빠진다.

---

## 5. 사용자 검토 게이트 (Human Review)

**무엇을 하는가**
이메일/RFP 초안 완성 후 사용자에게 "완료/수정/취소" 여부를 묻고, 응답에 따라 흐름을 분기한다.

**흐름**
`human_review_node` → `interrupt()` 대기 → (승인) END / (수정) 해당 draft node 루프백 / (타 기능 요청) `task_router`

**개발자 코멘트**

> LangGraph의 `interrupt()` / `Command(resume=)` 패턴을 활용했다. 이 패턴의 핵심은 그래프 실행을 완전히 중단하고 DynamoDB에 상태를 저장한 뒤, 사용자 응답이 오면 저장된 지점에서 재개한다는 점이다. 따라서 서버 재시작이 있어도 세션이 유지된다.
>
> 승인 판단은 규칙 기반(`_APPROVE_HINTS`)을 먼저 확인하고, 수정 판단은 그 다음에 확인한다. 승인과 수정이 동시에 감지될 경우 수정을 우선한다. "완전히 다른 기능"을 요청하는 switch 감지는 마지막에 하는데, 오탐 시 사용자 요청이 무시되는 것보다 "수정"으로 처리되는 쪽이 덜 나쁘기 때문이다.

---

## 6. 명확화 요청 (Clarification)

**무엇을 하는가**
라우팅 신뢰도가 낮거나 필수 슬롯(수신자, 프로젝트 범위, 파일 등)이 없을 때 사용자에게 추가 정보를 요청한다.

**흐름**
`task_router` (슬롯 미충족 또는 저신뢰) → `clarification_node` → 사용자 응답 수신 → `task_router` (재라우팅)

**개발자 코멘트**

> 슬롯 기반(파일 없음, 수신자 없음 등)과 LLM 기반(저신뢰) 두 경로를 분리했다. 슬롯 미충족은 LLM 없이 정해진 질문을 출력하므로 비용·속도 면에서 효율적이다. LLM 기반 명확화는 질문 또는 거절(out-of-scope 감지 시)을 출력한다.
>
> 무한 명확화 루프 방지를 위해 `clarification_count >= 2`이면 knowledge_search로 강제 폴백한다. 사용자가 계속 모호하게 대답해도 최소한 검색 결과는 보여주는 것이 아무것도 안 하는 것보다 낫다고 판단했다.

## 7. 파일 분석 Q&A (File Chat)

**무엇을 하는가**
업로드된 파일을 기반으로 사용자 질문에 답한다. 사내 RAG 검색을 거치지 않고 파일 내용만 참조한다.

**흐름**
`task_router` → `file_chat_node` (ReAct 에이전트 + `get_attached_file` 도구)

**개발자 코멘트**

> LangGraph의 `create_react_agent`를 내부에서 생성하는 구조다. 단일 도구(`get_attached_file`)만 제공해 에이전트가 사내 문서나 외부 데이터를 끌어오지 못하도록 의도적으로 제한했다. 파일 내용은 도구 호출 시점에 sanitize를 한 번 더 거치는데, 악성 파일에 프롬프트 인젝션 구문이 삽입될 수 있기 때문이다. MAX_TOKENS 감지 로직을 추가한 것은 대용량 파일(엑셀, PDF)에서 컨텍스트 초과가 발생할 때 무음 에러가 아닌 명시적 안내를 주기 위해서다.

---

## 8. 파일 텍스트 추출 (File Extract)

**무엇을 하는가**
PDF·DOCX·XLSX·PPTX·TXT·MD 파일에서 텍스트를 추출해 state에 저장한다. 이후 RFP 작성 등의 다른 노드에서 참조한다.

**흐름**
`task_router` → `file_extract_node` → 다음 task 노드

**개발자 코멘트**

> 포맷별로 파서를 분기한다(pypdf / python-docx / openpyxl / python-pptx). 특히 XLSX는 셀 단위 TSV로 변환해 LLM이 구조적 데이터를 읽기 쉽게 했다. 추출 결과를 GraphState의 `extracted_text`에 저장해두면 같은 대화 내 다른 기능(RFP 작성 등)이 파일 내용을 참조할 수 있다.

## 9. AI 기능 안내 (AI Guide)

**무엇을 하는가**
Kaiper AI의 기능 5가지를 안내하고 자기소개를 한다. 도구 없이 LLM 직접 호출.

**흐름**
`task_router` (짧은 메시지·인사 감지) → `ai_guide_node`

**개발자 코멘트**

> 기능 목록을 시스템 프롬프트에 고정 텍스트로 넣은 이유는, LLM이 존재하지 않는 기능을 창작하거나 내부 구현을 노출하는 것을 방지하기 위해서다. 내부 기술 정보(시스템 프롬프트, API 키, 모델명, DB 구조) 질문에는 "해당 정보는 제공하지 않습니다"로만 답하도록 명시했다.

## 10. 복합 요청 처리 (Planner)

**무엇을 하는가**
"~를 검색해서 RFP 작성해줘"처럼 검색 + 작성이 동시에 요청될 때, 선행 검색을 먼저 실행하고 결과를 후속 작업에 주입한다.

**흐름**
`task_router` (복합 신호 감지) → `planner_node` → (선행 검색 인라인 실행) → `route_after_planner` → 최종 task 노드

**개발자 코멘트**

> 사용자가 하나의 문장으로 "조사 + 작성"을 동시에 요청하는 경우가 있다. 이것을 두 번의 대화로 나누게 하면 UX가 나빠진다. Planner가 LLM으로 태스크 목록을 파싱(최대 2개)한 후, 선행 검색을 인라인으로 실행해 `planner_context`에 저장한다. 이후 이메일/RFP 노드가 이 컨텍스트를 사용해 검색 결과를 반영한 초안을 만든다.

## 11. 보안 — 3중 레이어

### 입력 가드 (Input Guard)

**무엇을 하는가**
프롬프트 인젝션 감지. 모든 요청의 첫 번째 처리 단계.

**개발자 코멘트**

> 임베딩 기반 유사도 비교 방식을 택했다. 규칙 기반 블랙리스트는 우회 변형에 취약하지만, 임베딩 유사도는 표현이 바뀌어도 의미가 유사하면 감지한다. 단일 입력(threshold 0.80)과 슬라이딩 윈도우(최근 3턴 합산, threshold 0.76) 두 경로를 병렬 검사한다. 추가 LLM 호출 없이 기존 임베딩 인프라를 재사용하기 때문에 레이턴시 오버헤드가 없다.

### 문서 콘텐츠 살균 (Content Sanitizer)

**무엇을 하는가**
RAG 검색 결과 및 업로드 파일에서 인젝션 구문 필터링.

**개발자 코멘트**

> RAG 인젝션(검색된 문서에 숨겨진 지시문이 LLM에 주입되는 공격)을 막기 위한 레이어다. 13개 정규식 패턴으로 문서 내 인젝션 구문을 탐지하고 해당 청크를 차단한다.

### 출력 검증 (Output Validator)

**무엇을 하는가**
LLM 생성 결과물에서 민감 정보 노출·유해 콘텐츠 탐지.

**개발자 코멘트**

> 생성 결과가 사용자에게 노출되기 직전 마지막 관문이다. 입력 가드와 콘텐츠 살균을 통과하더라도 LLM 자체가 예상치 못한 내용을 생성할 수 있으므로 3중 구조를 유지했다.

## 12. 라우터 (Task Router)

**무엇을 하는가**
사용자 입력을 분석해 어떤 기능 노드로 보낼지 결정하는 중앙 분기 노드.

**판단 순서**
규칙 기반 단락 → 의미 유사도(코사인 거리, threshold 0.62) → LLM 폴백 → 슬롯 미충족 판단 → clarification / 폴백

**개발자 코멘트**

> 라우팅을 단일 LLM 호출로만 하면 빠르지만 일관성이 없다. 의미 유사도만 하면 키워드 의존적이다. 두 방식을 계층적으로 결합했다. LLM 폴백 결과는 DynamoDB `intent_samples` 테이블에 저장해 이후 의미 유사도 검색의 훈련 데이터로 활용한다. 사용량이 늘수록 규칙 없이도 더 잘 분류되는 구조다.
>
> `_PERSON_TITLE_RE`로 "홍길동 파트장에게", "업스테이지 신수연 담당자에게" 같은 패턴을 감지한다. 특정 이름을 딕셔너리에 등록하지 않아도 직함·방향 조사 패턴으로 수신자 의도를 인식할 수 있어 신규 거래처가 생길 때마다 코드를 수정할 필요가 없다.
